Semantics and Standards

Speaking of logic.

Thinking about logical order and such sparks a whole thought process on the way things should be, the entire concept of html5, for me, weighs heavy on semantics, standards, etc.

There is a specific intention for each of the new elements and attributes, also a revisited or renewed intention for some of the old ones. Take this article for example. The article element wasn't meant to be just thrown into your markup and filled with just any content it's meant to contain a group of other elements new and old that make up exactly that an article. Okay well what should an article contain, conceptually. Look at other mediums that use articles like magazines, newspapers, etc. Well your standard article in these contexts is made up of a few things, you have a header with a title and sub-title, oh and by the way the role attribute intended for sections firstly isn't just for sections, secondly it isn't just for safari's reader function, we'll get back to roles later, next you might have a synopsis followed by the body which is not meant to be just text it could have anything in it, especially on the web, audio, video, images, etc., in a more technical setting you might have references, and meta data. With paper media you also might have what html5 calls an aside and yes while aside were meant to be positioned to the left or right of their siblings that doesn't make them good for sidebars in website layouts, thats just one example from a list apart, i think, that got used literally, the more logical application is within articles for an authors bio or for figures like in text books and technical documents, they are good for side notes, but most importantly they are like articles and they are meant to contain sub-sections eg a title, body, footer. Lets dig deeper and talk about standards and not just semantics. Most importantly you should come up with a set of rules for how you write your markup, and i break my own rules all the time even in writing this i'm thinking about all the rules i'm breaking just in this document alone, okay so lets go all the way back and look at the big picture, a website is should be a hierarchy of webpages, a webpage has should have it's own internal hierarchy as well, and i should add that i'm not talking in linear terms either like the header, content and footer regions, thats the structure of the template, visualize it as layers in 3d space, okay first you have the template layer, thats the frame around the content area, the content area is where the actual document lives, and in the document you have articles and media and so on, and the articles have content and the articles content has content or references to other articles or documents or pages or sites and so on, at the smallest level each component is an element with attributes and content and i'm constantly reminded of an article i read on nettuts about well formed or well written stylesheets in the back of mind and from that while developing this document i'm really getting into the way things should be, okay so you have an element with attributes and content and not every document has both attributes or content simultaneously you might have a paragraph with no class, role, id whatever but lots of content or you might have an image which is self closing and has no content but has lots of attributes, eg file source, title, whatever is neccasary to get it off the server and into the document, you could have a link with no attributes or content just a simple <a/> then style it with css and give it an action with js. That's not the point, what i'm trying to get accross is that it's open to your interpretation of what should be at that moment, but you need to give yourself a markup writing set of guidelines for standard situations in order to write well formed documents for yourself and if you break it down to the most specific level the end result will be highly desireable, eg in my case, i've decided for markup purposes that all elements with attributes need to have those attributes sorted in a logical order, lets say you have a div with an id, a class and a role defined, what order should those attributes go in? Exactly that id > class > role now even the content of the class attribute should be sorted if need be so for me the order is the most logical one, again, vague to specific, like this, in this document there is a div, it's got siblings because it's part of a set of columns i'm using the grid system so i need to set the number of columns that it spans across the grid in this case it's four out of twelve so the first classname in the class attribute is fourcol and i have style properties that manipulate the first or last columns for layout purposes so the next classname is first cause it's the first column &emdash;more specific, which fourcol is it, the first one, now the last classname is box and columns with the class box have additional more specific style properties. Let's talk about html5 and template markup, the new elements, header, section, aside and footer, weren't meant for making templates more semantic in terms of DOM it was never meant to be, imho, body > header + section + (aside + aside) + footer. A well formed template should use divs till you get to the div containing the content and the content should be made then made up of sections with roles and those sections should have articles with all the subsequent elements needed to build a well formed article and so on.
